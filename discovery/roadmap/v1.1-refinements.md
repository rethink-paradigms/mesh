# Mesh v1.1 Refinements

## Summary

This roadmap captures 16 design decisions (DE1-DE16) that refine Mesh's v1.1 scope. These decisions address the gap between the original design intent (D1-D10) and the concrete implementation plan. The refines center on three themes: substrate strategy (all providers equal, Fly adapter clarified), architecture (plugins over built-ins, single daemon, extension interfaces), and scope containment (static scheduler, no filesystem merge, deferred Pulumi). Together they lock in a v1.1 that is plugin-centric, user-configured, and deliberately minimal.

Status: 15 accepted, 1 discarded. All decisions dated April 29, 2026.

---

## Decisions

### DE1: All sandbox providers equal -- no primary designation

**Status**: accepted **Date**: 2026-04-29

All sandbox substrate providers are evaluated on the same criteria: Docker/OCI compatibility, filesystem persistence, suspend/resume, Tailscale integration, billing model, and runtime limits. No single provider is designated primary. Fly Machines, E2B, Modal, Cloudflare Containers, and Daytona were evaluated against these criteria. E2B, Modal, and Cloudflare Containers were excluded based on filesystem and runtime constraints. Users select their substrate explicitly in config; there is no auto-scheduling layer.

### DE2: Static scheduler config for v1.1 -- no auto-scheduling

**Status**: accepted **Date**: 2026-04-29

The v1.1 scheduler is purely static configuration. Users specify which substrate each body runs on in `~/.mesh/config.yaml`. There is no automatic scheduler. Auto-scheduling adds cost modeling, capacity queries, and idle-detection complexity without proven demand. Early adopters know where they want their agents. Idle detection is especially tricky -- zero CPU can mean network waits, not idle. A plugin-based scheduler is a v2.0 consideration.

### DE3: Filesystem delta merge deferred to v2.0

**Status**: accepted **Date**: 2026-04-29

Clone-and-merge (clone body, run on sandbox, merge changes back to parent) is deferred beyond v1.1. The persona requirement (A4) only calls for optional merge. Clone-and-run without merge satisfies the primary burst compute use case: collect output artifacts and destroy the clone. Three-way merge semantics add enormous complexity for no proven demand. v1.1 supports clone from snapshot, run on sandbox, collect output artifacts, and destroy. No filesystem merge.

### DE4: Pulumi unsuitable for Mesh -- use OpenAPI + SDK + template pipeline

**Status**: accepted **Date**: 2026-04-29

This decision re-evaluates D6 (provider integrations as AI-generated plugins via Pulumi). The plugin architecture from D6 is correct, but Pulumi is not viable as the generation mechanism. Pulumi generates infrastructure-as-code programs, not SubstrateAdapter plugin code. The correct approach is an OpenAPI + oapi-codegen v2 pipeline: generate typed Go clients from provider OpenAPI specs, then apply an AI mapping layer to bridge to the SubstrateAdapter interface. For v1.1, the Fly Machines and Docker adapters are hand-written reference implementations. This decision supersedes the generation method in D6.

### DE5: Git-based plugin distribution for v1.1, binary registry deferred

**Status**: accepted **Date**: 2026-04-29

Plugins are distributed via Git repositories. Users install with `mesh plugin install github.com/user/mesh-substrate-fly`. The daemon clones the repo, runs `go build`, and loads the binary. This aligns with Go ecosystem conventions, requires no central registry, and lets users pin commits or tags. Plugin naming convention is `mesh-substrate-<name>`. Multiple plugin versions can coexist. Signing and a binary registry are deferred to v2.0. Cross-language plugins are possible via gRPC but not prioritized.

### DE6: Skills live in daemon core, MCP resources, and agent-side -- a mixed model

**Status**: accepted **Date**: 2026-04-29

Skills are classified into three tiers. Tier 1 (daemon core): body lifecycle operations compiled into the daemon binary. Tier 2 (MCP resources): operational skills like pack-vm and garbage-collect, served as MCP resources from separate Go packages. Tier 3 (agent-side compositions): user-level skills that compose Mesh primitives, living in the AI agent's skill directory. This keeps the daemon small, enables extensibility, and gives users a clear boundary for custom workflows.

### DE7: Production hardening priorities for v1.1

**Status**: accepted **Date**: 2026-04-29

Ranked hardening priorities. P1 (critical): snapshot corruption recovery with SHA-256 verification and dual copies, daemon crash resilience with SQLite WAL and reconciliation loop. P2 (high): secrets management via env-var references, PID file validation. P3 (medium): networking warning without Tailscale, daemon stop-restart upgrades. P4 (deferred): zero-downtime upgrades, daemon discovery after machine death, body-to-body communication.

### DE8: Docker adapter is a plugin, not built-in

**Status**: accepted **Date**: 2026-04-29

Resolves EQ1. Docker becomes a plugin consistent with Nomad's plugin model. The only built-in adapter is the local filesystem adapter (for A5 developer agent use). Making Docker a plugin forces the plugin loading system to be robust from day one. Users who don't need Docker don't load it. The daemon startup sequence becomes config, store, plugins, MCP server. If the Docker plugin fails to load, the daemon still starts with a warning.

### DE9: Single daemon architecture is correct for v1.1

**Status**: accepted **Date**: 2026-04-29

Resolves EQ2. A single Mesh daemon serves as the control plane for all bodies across all substrates. It maintains the SQLite store, manages MCP connections, and orchestrates lifecycle. Bodies run on their respective substrates, not in the daemon process. The daemon is a lightweight control plane (~50MB RAM). It is a single point of control but not a single point of failure for running bodies -- they continue on their substrates if the daemon stops.

### DE10: SQLite store backup is manual dump + reconciliation

**Status**: accepted **Date**: 2026-04-29

Resolves EQ4. The SQLite store uses WAL mode for crash resilience. Manual backup via `mesh db backup` writes a SQL dump. No automatic backup in v1.1 because the store holds metadata only (body IDs, states, snapshot locations), not body filesystems. Loss means losing track of bodies, not the bodies themselves. Recovery works by rescanning substrates and rebuilding via reconciliation.

### DE11: Daemon upgrades: stop-restart, bodies unaffected

**Status**: accepted **Date**: 2026-04-29

Resolves EQ3. Upgrades follow a stop-restart model: SIGTERM the daemon, wait up to 30s for graceful shutdown, start the new version. Bodies on substrates are unaffected -- they keep running. The daemon reconciles on startup by rescanning substrates and matching instances to store entries. Rolling and zero-downtime upgrades are deferred to v2.0.

### DE12: Migration testing uses layered approach

**Status**: accepted **Date**: 2026-04-29

Resolves EQ5. Four testing layers. Layer 1: unit tests with mock SubstrateAdapter interfaces. Layer 2: Docker-to-Docker integration tests on real Docker. Layer 3: manual Docker-to-Fly tests with real Fly credentials. Layer 4: CI/CD runs Docker-to-Docker in GitHub Actions. Fly and Nomad migration tests remain manual until CI-accessible test clusters exist.

### DE13: Fly Machines adapter -- accept no-filesystem-export limitation

**Status**: accepted **Date**: 2026-04-29

An API spike revealed Fly Machines has no filesystem tarball export API. Volumes are host-tied and there is no stable REST exec API. The adapter strategy adapts: persistent body data goes on Fly Volumes mounted at /data, exec uses the flyctl CLI wrapper, body migration from sandbox requires self-export from within the container. The fleet pool (Docker/Nomad) remains the primary body storage location. The sandbox pool handles ephemeral work only (burst compute, clone-and-run).

### DE14: database/sql pattern as Phase 1 plugin architecture

**Status**: accepted **Date**: 2026-04-29

The core SubstrateAdapter interface stays minimal (6 verbs). Optional capabilities are exposed via extension interfaces (SubstrateLister, SubstrateLogger, SubstrateExecutor, etc.) that plugins can optionally implement. Core checks capability availability via Go type assertion before calling extension methods. This matches the database/sql driver pattern from the Go ecosystem -- no bloated interface, compile-time safety, clear capability discovery.

### DE15: oapi-codegen v2 as primary codegen tool

**Status**: accepted **Date**: 2026-04-29

oapi-codegen v2 is the primary tool for generating typed Go clients from provider OpenAPI specs. It generates idiomatic Go code without Java dependencies, supports OpenAPI 3.1, and produces output compatible with the AI mapping layer. Alternatives (swagger-codegen, openapi-generator, oapi-codegen v1, custom templates) were rejected for various reasons including Java dependency, deprecated status, or maintenance burden.

### DE16: Extension interfaces for optional capabilities, not ErrNotImplemented

**Status**: accepted **Date**: 2026-04-29

Optional capabilities use extension interfaces with type assertion, not an ErrNotImplemented pattern. If a plugin does not implement SubstrateLister, the list feature is simply unavailable. This avoids silent failures from ignored ErrNotImplemented returns, provides compile-time safety, and matches Go best practices. The fat-interface-with-ErrNotImplemented anti-pattern is explicitly rejected.

## Cross-Cutting Impact

### DE4 supersedes D6 (Provider integrations)

DE4 replaces the Pulumi-based generation approach from D6 with an OpenAPI + SDK + template pipeline. The plugin architecture from D6 remains intact -- provider integrations are still plugins, still generated, and still outside core. What changes is the generation method: OpenAPI specs feed oapi-codegen v2 instead of Pulumi programs feeding an LLM bridge. For v1.1, the first two adapters (Docker and Fly Machines) are hand-written reference implementations, establishing the interface contract that the codegen pipeline targets.

### DE8 impacts plugin architecture (Docker as plugin)

DE8 extends the plugin-first principle to Docker. Previously Docker was a built-in adapter while Nomad was a plugin. This was inconsistent. Making Docker load through the same plugin mechanism as every other substrate forces the loading system to be production-ready from day one. It also means the daemon can start without Docker -- useful for users who only use Fly Machines or Nomad. The daemon startup sequence becomes deterministic: config, store, plugins, MCP server. Plugin loading failures are non-fatal (daemon starts with warnings).

### DE1 impacts provider parity (All providers equal)

DE1 establishes that no provider gets special treatment in the core. All candidates are evaluated against the same criteria. This prevents architectural coupling to any single provider's API quirks. The practical impact is that body storage is provider-agnostic -- snapshots are portable tarballs, not provider-specific volumes. DE13 (Fly Machines limitation) demonstrates this principle in action: rather than treating Fly as primary and working around its API gaps with special cases, the adapter accepts the limitations and routes around them with self-export and explicit volume management.

### DE2 impacts scheduler design (Static configuration)

DE2 removes the auto-scheduler from v1.1 entirely. This is a significant scope reduction from the original vision where substrate selection was a core or plugin concern. The user becomes the scheduler via config file. This keeps the daemon small, avoids baking in an idle-detection model that would likely be wrong, and defers the hard problem (cost-aware, capacity-aware scheduling) to when demand is proven. The static model also means no provider pricing APIs, no capacity queries, and no automatic migration -- users move bodies explicitly.

## Theme 4: Architecture Correction (DE17-DE20)

Architectural review revealed the single SubstrateAdapter conflated orchestration with provisioning. These decisions correct the foundation.

| ID | Decision | Impact |
|----|----------|--------|
| DE17 | Two-pool adapter architecture | Replaces SubstrateAdapter with OrchestratorAdapter + ProvisionerAdapter |
| DE18 | No direct runtime access | Mesh talks to orchestrators only. Docker adapter removed. |
| DE19 | Cloud-init bootstrap pattern | Orchestrator generates scripts, provisioner passes as user-data |
| DE20 | Local support deferred | v1 focuses on fleet + sandbox. Local needs separate research. |

**Cross-cutting impact:**
- DE8 (Docker as plugin) → REJECTED, superseded by DE17/DE18
- DE14 (database/sql pattern) → reframed for both pools
- DE16 (extension interfaces) → reframed for both adapter types
- DE1, DE2, DE13 → reframed for two-pool model
