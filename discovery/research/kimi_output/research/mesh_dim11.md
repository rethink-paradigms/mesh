# Dimension 11: Go Language Patterns for Adapter Generation

**Research Date**: 2025-07-15
**Scope**: Go patterns that make AI-generated adapters reliable, with prescriptive best practices

---

## 1. Executive Summary

- **Compile-time interface checks** (`var _ Interface = (*Type)(nil)`) are the standard Go pattern and should always be included in generated adapters. They catch signature mismatches at build time and serve as self-documenting code.[^639^][^640^]
- **Optional capabilities** should use the **extension interface pattern** (base interface + separate optional interfaces + type assertion), as formalized by the Go team in `io/fs`.[^677^] This is the same pattern `database/sql/driver` has used since Go 1.10 for `Pinger`, `SessionResetter`, `ExecerContext`, etc.[^377^]
- **`context.Context` must be the first parameter** of every adapter method, named `ctx`. Never store contexts in structs. The generator should enforce this mechanically.[^718^][^745^]
- **Error wrapping at adapter boundaries** should use sentinel errors (e.g., `ErrNotSupported`) rather than exposing provider-specific errors. Wrap with `fmt.Errorf("...: %w", err)` only when callers need to inspect the underlying error.[^637^][^766^]
- **The `go:generate` directive** is the idiomatic way to trigger adapter generation. Generated files must include a `// Code generated by TOOL; DO NOT EDIT.` header and be checked into version control. CI should verify generated code is up-to-date.[^631^][^638^]
- **Generics are unnecessary** for adapter generation with ~8 concrete methods. No major Go project (database/sql, io/fs, Go CDK, HashiCorp go-plugin) uses generics for adapter mapping.[^706^]
- **Build tags** are useful for platform-specific or experimental provider code but are overkill for simple adapter selection. Prefer runtime registration (like database/sql) or separate packages over build tags for provider variants.[^703^]
- **The "portable type + driver interface" pattern** (from database/sql and Go CDK) is the most robust architecture: a concrete struct exposes the user-facing API, while a minimal driver interface is implemented per-provider.[^341^][^713^]

## 2. Detailed Findings

### 2.1 Interface Satisfaction: Compile-Time Checks

#### 2.1.1 The Standard Pattern

Go's standard mechanism for compile-time interface satisfaction is the blank identifier assignment:

```go
var _ InterfaceName = (*TypeName)(nil)
```

This pattern is endorsed by the Go Authors in the official FAQ:[^639^]

> You can ask the compiler to check that the type `T` implements the interface `I` by attempting an assignment using the zero value for `T` or pointer to `T`, as appropriate.

Claim: The `(*T)(nil)` form avoids memory allocation and is preferred over `T{}` when methods have pointer receivers.[^639^]
Source: Stack Overflow / Go FAQ
URL: https://stackoverflow.com/questions/10498547/ensure-a-type-implements-an-interface-at-compile-time-in-go
Date: 2022-09-07
Excerpt: "`(*T)(nil)` creates an uninitialized pointer to a value of type `T` by converting `nil` to `*T`. See golang interface compliance compile type check. This avoids allocation of memory for an empty struct as you'd get with `&T{}` or `new(T)`."
Context: Community-curated FAQ answer quoting the Go Authors
Confidence: high

#### 2.1.2 When to Use Value vs. Pointer Receivers

The choice between `TypeName{}` and `(*TypeName)(nil)` depends on method receivers:[^627^]

- **Value receivers**: Either form works.
- **Pointer receivers**: Must use `(*TypeName)(nil)`.

Claim: The generator should always emit `var _ SubstrateAdapter = (*ProviderAdapter)(nil)` because adapters typically need mutable state and therefore use pointer receivers.
Source: Dev.to / Kittipat.po
URL: https://dev.to/kittipat1413/checking-if-a-type-satisfies-an-interface-in-go-432n
Date: 2024-10-30
Excerpt: "If `TypeName` implements any methods with pointer receivers (e.g., func `(t *TypeName) Method()`), you must use `(*TypeName)(nil)`."
Context: Tutorial article explaining compile-time checks
Confidence: high

#### 2.1.3 Should the Generator Always Include This?

Yes. The compile-time check serves two purposes: it prevents runtime panics from incomplete implementations, and it acts as explicit documentation that the struct is intended to satisfy the interface.[^629^]

Claim: If the Go code being written is intended to be used as a library by others, then you'll likely want to use this type of assertion as the library may not be using the interface methods on the struct, yet those methods are expected to be defined on the struct by those using the library.
Source: Erica Pisani's blog
URL: https://www.ericapisani.dev/avoiding-go-runtime-errors-with-interface-compliance-and-type-assertion-checks/
Date: 2025-02-13
Excerpt: "If the Go code being written is intended to be used as a library by others, then you'll likely want to use this type of assertion as the library may not be using the interface methods on the `struct`, yet those methods are expected to be defined on the `struct` by those using the library."
Context: Blog post about avoiding runtime errors
Confidence: high

**Recommendation**: The generator must emit compile-time interface checks for every generated adapter. This is non-negotiable for reliability.

### 2.2 Optional Methods: The Extension Interface Pattern

#### 2.2.1 The Problem

Go interfaces do not support optional methods. All methods in an interface are mandatory. When a substrate (like a cloud provider or database) supports only a subset of capabilities, the adapter needs a pattern for optional functionality.

#### 2.2.2 Pattern A: Extension Interfaces (Recommended)

The Go team formalized this pattern in the `io/fs` proposal (Go 1.16) under the name **extension interfaces**:[^677^]

> An extension interface embeds a base interface and adds one or more extra methods, as a way of specifying optional functionality that may be provided by an instance of the base interface. An extension interface is named by prefixing the base interface name with the new method: a `File` with `ReadDir` is a `ReadDirFile`.

```go
type File interface {
    Stat() (FileInfo, error)
    Read([]byte) (int, error)
    Close() error
}

// Extension interface: optional ReadDir capability
type ReadDirFile interface {
    File
    ReadDir(n int) ([]DirEntry, error)
}
```

The consumer uses a type assertion with fallback:

```go
func ReadDir(fsys FS, name string) ([]DirEntry, error) {
    if fsys, ok := fsys.(ReadDirFS); ok {
        return fsys.ReadDir(name)
    }
    // fallback using Open + ReadDirFile
}
```

Claim: This pattern is an old Go pattern that we've never named before but that we suggest calling an extension interface.
Source: Go Proposal — File System Interfaces for Go (draft-iofs.md)
URL: https://go.googlesource.com/proposal/+/master/design/draft-iofs.md
Date: 2020 (draft for Go 1.16)
Excerpt: "An extension interface is named by prefixing the base interface name with the new method: a File with ReadDir is a ReadDirFile."
Context: Official Go design document for io/fs package
Confidence: high

#### 2.2.3 Pattern B: Sentinel Error Return (Avoid)

Some code returns `ErrNotSupported` from required methods when a capability is unavailable. This is less idiomatic because:
- Callers must invoke the method to discover it's unsupported.
- There's no way to pre-check capability without a side-effecting call.
- It pollutes the method contract with a non-error condition.

The Go team explicitly rejected this for `io/fs` in favor of extension interfaces.[^684^]

Claim: A few people raised concerns about the use of optional interfaces, but those are an established pattern in Go that we understand how to use well.
Source: golang/go Issue #41190
URL: https://github.com/golang/go/issues/41190
Date: 2020-09-02
Excerpt: "A few people raised concerns about the use of optional interfaces, but those are an established pattern in Go that we understand how to use well (informed in part by some earlier mistakes, such as optional interface like http.Hijacker with methods that cannot return an error to signal failure/unavailability)."
Context: Russ Cox's acceptance of the io/fs proposal
Confidence: high

#### 2.2.4 Pattern C: Capabilities Struct (Niche)

A capabilities struct can be useful when many binary yes/no capabilities exist:

```go
type Capabilities struct {
    SupportsFilesystem bool
    SupportsNetwork  bool
}

func (a *Adapter) Capabilities() Capabilities
```

This pattern appears in some plugin systems but is not standard in Go. It adds API surface area without the type safety of extension interfaces.

#### 2.2.5 What database/sql Uses

The `database/sql/driver` package is the canonical example of extension interfaces in Go:[^377^]

- `Pinger`, `SessionResetter`, `Validator` — all optional on `Conn`
- `ExecerContext`, `QueryerContext`, `ConnBeginTx` — context-aware versions
- `NamedValueChecker` — custom data type support
- `RowsNextResultSet` — multiple result sets
- `RowsColumnTypeScanType`, `RowsColumnTypeDatabaseTypeName`, etc. — metadata

Claim: If named parameters or context are supported, the driver's Conn should implement: ExecerContext, QueryerContext, ConnPrepareContext, and ConnBeginTx.
Source: database/sql/driver package docs
URL: https://pkg.go.dev/database/sql/driver
Date: Current
Excerpt: "All Conn implementations should implement the following interfaces: Pinger, SessionResetter, and Validator."
Context: Official Go standard library documentation
Confidence: high

#### 2.2.6 What Go CDK Uses

The Go CDK (gocloud.dev) follows the same pattern as database/sql — portable types with driver interfaces:[^713^]

```go
// Portable type (concrete struct)
type Bucket struct {
    b driver.Bucket
}

// Driver interface (minimal)
type Bucket interface {
    ErrorCode(error) gcerrors.ErrorCode
    As(i any) bool
    Attributes(ctx context.Context, key string) (*Attributes, error)
    // ...
}
```

Claim: When new operations on the driver are added as new optional interfaces, the portable type can hide the need for type-assertions from the user.
Source: gocloud.dev — Structuring Portable Code
URL: https://gocloud.dev/concepts/structure/
Date: Current
Excerpt: "When new operations on the driver are added as new optional interfaces, the portable type can hide the need for type-assertions from the user."
Context: Official Go CDK design documentation
Confidence: high

#### 2.2.7 What HashiCorp go-plugin Uses

HashiCorp's go-plugin uses RPC-based plugin interfaces but does not use extension interfaces — instead, the plugin author defines a single interface and the framework generates RPC stubs for it.[^704^] The framework itself does not deal with optional methods at the interface level.

**Recommendation**: For Mesh adapter generation, use **extension interfaces** (Pattern A) for optional capabilities. Name them `{Method}{BaseInterface}` (e.g., `FilesystemAdapter` for an adapter with filesystem methods). The generator should emit both the base adapter and optional extension interfaces. The portable type (concrete struct) should hide type assertions from callers.

### 2.3 Context Propagation

#### 2.3.1 The Go Context Contract

The `context` package documentation establishes strict rules:[^718^]

> Do not store Contexts inside a struct type; instead, pass a Context explicitly to each function that needs it. The Context should be the first parameter, typically named ctx.

Claim: Programs that use Contexts should follow these rules to keep interfaces consistent across packages and enable static analysis tools to check context propagation.
Source: context package documentation
URL: https://pkg.go.dev/context
Date: Current
Excerpt: "Do not store Contexts inside a struct type; instead, pass a Context explicitly to each function that needs it."
Context: Official Go standard library documentation
Confidence: high

#### 2.3.2 Generator Enforcement

The generator must enforce:
1. Every adapter method takes `ctx context.Context` as its **first** parameter.
2. The parameter is named `ctx` (enables linter compatibility).
3. The context is passed through to every underlying provider call.
4. Contexts are never stored in adapter struct fields.

Claim: When a function takes a context parameter, that context should only be used for the duration of the call, not after it returns.
Source: Horace / Context Control in Go
URL: https://zenhorace.dev/blog/context-control-go/
Date: 2024-02-03
Excerpt: "When a function takes a context parameter, that context should only be used for the duration of the call, not after it returns."
Context: Blog post on context best practices
Confidence: high

#### 2.3.3 Context Cancellation Gotchas During Migration

When migrating from non-context-aware code to context-aware adapters, several gotchas exist:[^740^][^747^]

1. **Long-running operations**: If the underlying provider doesn't support context cancellation, the adapter must either spawn a goroutine to watch `ctx.Done()` or document that cancellation is best-effort.
2. **Goroutine leaks**: Adapters that spawn goroutines must ensure they exit when the context is cancelled.
3. **Nested contexts**: Never call `context.Background()` deep in the call stack — always propagate the received context.[^740^]
4. **Cancel function leaks**: Always call `defer cancel()` after creating a derived context.

Claim: If your goroutine blocks on a channel or other operation, it will not notice cancellation unless you also select on ctx.Done().
Source: OneUptime blog / Gracefully Cancel Long-Running Goroutines
URL: https://oneuptime.com/blog/post/2026-01-25-gracefully-cancel-goroutines-context-go/view
Date: 2026-01-25
Excerpt: "If your goroutine blocks on a channel or other operation, it will not notice cancellation unless you also select on ctx.Done()."
Context: Practical guide to context cancellation
Confidence: high

**Recommendation**: The generator should produce code that accepts `ctx context.Context` as the first parameter of every method and passes it through unmodified to provider calls. If a provider API doesn't accept context, the adapter should document this limitation in a comment.

### 2.4 Error Wrapping

#### 2.4.1 The Go 1.13+ Standard

Go 1.13 introduced `%w` in `fmt.Errorf` to preserve error chains:[^637^]

```go
return fmt.Errorf("decompress %v: %w", name, err)
```

This enables `errors.Is()` and `errors.As()` to traverse the chain.

Claim: Wrapping an error with `%w` makes it available to `errors.Is` and `errors.As`.
Source: Go Blog — Working with Errors in Go 1.13
URL: https://go.dev/blog/go1.13-errors
Date: 2019-10-17
Excerpt: "Wrapping an error with `%w` makes it available to `errors.Is` and `errors.As`."
Context: Official Go blog post
Confidence: high

#### 2.4.2 Whether to Wrap at Adapter Boundaries

The Go blog provides clear guidance:[^637^]

> Wrap an error to expose it to callers. Do not wrap an error when doing so would expose implementation details.

For adapters:
- **Wrap with `%w`** when the caller provided the resource (e.g., a reader, writer) and should be able to inspect errors from it.
- **Do NOT wrap with `%w`** when exposing a provider-specific error would couple callers to that provider.
- **Translate to sentinel errors** at package boundaries instead.[^766^]

Claim: At points where your system interacts with external systems like RPC, IPC, or storage, it's often better to translate domain-specific errors into a standardized error space rather than simply wrapping the raw underlying error with `%w`.
Source: Redowan's Reflections / Go errors: to wrap or not to wrap?
URL: https://rednafi.com/go/to-wrap-or-not-to-wrap/
Date: 2026-03-07
Excerpt: "At points where your system interacts with external systems like RPC, IPC, or storage, it's often better to translate domain-specific errors into a standardized error space (e.g., gRPC status codes) rather than simply wrapping the raw underlying error with `%w`."
Context: Comprehensive analysis of error wrapping practices
Confidence: high

#### 2.4.3 Sentinel Errors vs. Typed Errors

For adapter generation, the recommended pattern is:

```go
var (
    ErrNotSupported = errors.New("operation not supported by this substrate")
    ErrNotFound     = errors.New("resource not found")
    ErrTimeout      = errors.New("operation timed out")
)
```

These are **sentinel errors** — predefined package-level variables that callers can match with `errors.Is()`.[^758^]

Custom **typed errors** should be used when callers need structured data (e.g., the specific resource name that was not found).[^766^]

Claim: Sentinel errors remain valuable because they provide a shared signal across packages. Some conditions are best expressed as single fixed values.
Source: Medium / Error Values vs Sentinel Errors in Go
URL: https://medium.com/@AlexanderObregon/error-values-vs-sentinel-errors-in-go-a377c3d831d1
Date: 2025-11-06
Excerpt: "Sentinel errors remain valuable because they provide a shared signal across packages. Some conditions are best expressed as single fixed values."
Context: Analysis of sentinel error patterns
Confidence: high

**Recommendation**: The generator should:
1. Define standard sentinel errors for common substrate conditions (`ErrNotSupported`, `ErrNotFound`).
2. Wrap provider errors with `fmt.Errorf("operation: %w", err)` only when the caller needs to inspect the underlying error.
3. Use `fmt.Errorf("operation: %v", err)` (with `%v`) when hiding implementation details is preferred.
4. Never let raw provider errors escape without at least adding operation context.

### 2.5 Go Generate Directives

#### 2.5.1 Basic Usage

The `go generate` command scans for `//go:generate` comments and executes them:[^628^]

```go
//go:generate protoc --go_out=. myproto.proto
```

Claim: `go generate` only processes Go comments; it doesn't parse Go code. You can use multiple `//go:generate` lines in a single file. They'll run in order.
Source: Medium / Getting Started with go generate
URL: https://medium.com/@linz07m/getting-started-with-go-generate-automating-code-generation-in-go-8f6f1d94af4f
Date: 2025-05-04
Excerpt: "`go generate` only processes Go comments; it doesn't parse Go code. You can use multiple `//go:generate` lines in a single file. They'll run in order."
Context: Tutorial on go generate
Confidence: high

#### 2.5.2 Generated File Header

Every generated file must include:[^631^]

```go
// Code generated by mesh-gen; DO NOT EDIT.
```

This header tells editors, linters, and code review tools that the file is generated.

#### 2.5.3 Build Tags for Generator Code

Generator code (the tool that produces adapters) should use build tags to avoid being compiled with the main package:[^631^]

```go
//go:build ignore

package main

// Generator code here
```

#### 2.5.4 Build Process Integration

The standard approach is Makefile integration:[^631^]

```makefile
.PHONY: generate
generate:
    go generate ./...

.PHONY: build
build: generate
    go build ./...
```

#### 2.5.5 CI Verification

Generated code should be checked into version control, and CI should verify it's up-to-date:[^631^]

```yaml
# GitHub Actions
- name: Check generated code
  run: |
    go generate ./...
    git diff --exit-code
```

Claim: It is crucial to ensure that the "Go Generate" command is properly integrated into your build script or build system.
Source: Go Directives & Automation Tools Deep Dive
URL: https://nsddd.top/ai-technology/posts/directives-and-the-use-of-automation-tools/

Context: Guide on Go automation tools
Confidence: high

#### 2.5.6 Custom Generator Implementation

A custom adapter generator can parse Go source files using `go/ast` and `go/parser`, then emit generated code using `text/template` or direct file writing:[^631^][^759^]

```go
//go:generate go run cmd/mesh-gen/main.go
```

The generator receives environment variables:
- `$GOFILE` — current file name
- `$GOLINE` — line number of directive
- `$GOPACKAGE` — package name

Claim: The proper solution would be to generate code from ast, but ast is rather tedious to write, so most tools don't bother using it.
Source: Reddit / Easy typesafe code generation by converting go code to ast

Context: Community discussion on code generation techniques
Confidence: medium

**Recommendation**: 
1. Place `//go:generate` directives near the substrate interface definition.
2. Use `go run` for the generator so it doesn't need pre-installation.
3. Always include the `DO NOT EDIT` header.
4. Check generated files into version control.
5. Add a CI check that `go generate ./...` produces no diff.

### 2.6 Generics

#### 2.6.1 Are Generics Useful for Adapter Generation?

No. For a small, fixed set of methods (~8), generics provide no benefit and add complexity.

The Go standard library (database/sql, io/fs, net/http) does not use generics for adapter/plugin patterns. The Go CDK does not use generics for its driver interfaces. HashiCorp go-plugin does not use generics.

Claim: The Go CDK allows Go application developers to seamlessly deploy cloud applications on any combination of cloud providers. It does this by providing stable, idiomatic interfaces for common uses like storage and databases. Think database/sql for cloud products.
Source: Go CDK GitHub README
URL: https://github.com/google/go-cloud
Date: 2018-present
Excerpt: "The project works well with a code generator called Wire. It creates human-readable code that only imports the cloud SDKs for services you use."
Context: Go CDK project documentation
Confidence: high

#### 2.6.2 Where Generics Could Theoretically Help

If the Mesh architecture had a large number of method variants or needed type-safe collection operations, generics might help. But for the core adapter pattern:

- Interface satisfaction is static and checked at compile time.
- Method signatures are fixed (they must match the interface exactly).
- There's no type-family variation that would benefit from type parameters.

Claim: Implementation restriction: A union (with more than one term) cannot contain the predeclared identifier comparable or interfaces that specify methods.
Source: Mero's Blog / Constraining Complexity in the Generics Design
URL: https://blog.merovius.de/posts/2024-01-05_constraining_complexity/
Date: 2024-01-05
Excerpt: "cannot use fmt.Stringer in union (fmt.Stringer contains methods)"
Context: Analysis of Go generics constraints
Confidence: high

**Recommendation**: Do not use generics for adapter generation. Stick to standard interfaces and concrete types.

### 2.7 Build Tags

#### 2.7.1 Syntax (Go 1.17+)

Modern Go uses `//go:build` with boolean expressions:[^672^]

```go
//go:build linux && amd64
//go:build !(darwin && arm)
```

Legacy `// +build` comments are still supported but deprecated.

#### 2.7.2 File Suffixes vs. Build Tags

Go also supports conditional compilation via file suffixes:[^703^]

- `printer_linux.go` — compiled only on Linux
- `printer_darwin.go` — compiled only on macOS
- `printer_arm64.go` — compiled only on ARM64

File suffixes are simpler and preferred when the condition is purely OS/architecture-based. Build tags are needed for more complex logic (feature toggles, provider selection).

Claim: File naming with platform and architecture filters works without a build tag, though both can be combined for more precise control.
Source: Hoani.net / Go Conditional Compilation
URL: https://hoani.net/guides/software/golang/conditional-compilation
Date: 2022-06-05
Excerpt: "Sometimes if we are writing some kind of go driver, using file suffixes isn't enough."
Context: Guide on conditional compilation
Confidence: high

#### 2.7.3 When Build Tags Are Useful for Adapters

Build tags might be useful when:
- A provider requires CGO and must be excluded from pure-Go builds.
- A provider is experimental and should not be included in stable builds.
- Platform-specific substrates (e.g., Windows-specific filesystem features).

#### 2.7.4 When Build Tags Are Overkill

For normal provider selection, prefer:
- **Runtime registration** (like `database/sql`): `import _ "mesh/provider/aws"` registers the provider.
- **Separate packages**: Each provider in its own package; users import what they need.

Claim: Driver is magically registered during module import. It sounds like magic, isn't? And it turns out it is actually very simple. Function `init()` is package initializer, which can run arbitrary code before all other code.
Source: Miblog / Implement Sql Database Driver in 100 Lines of Go
URL: https://vyskocil.me/blog/implement-sql-database-driver-in-100-lines-of-go/
Date: 2019-02-18
Excerpt: "Driver is magically registered during module import... Function `init()` is package initializer."
Context: Tutorial on database/sql driver pattern
Confidence: high

**Recommendation**: Use runtime registration (`init()` + registry map) rather than build tags for provider selection. Build tags should be reserved for platform-specific or experimental provider code.

### 2.8 The Portable Type + Driver Pattern

#### 2.8.1 database/sql Architecture

The `database/sql` package separates the user-facing concrete type (`DB`) from the driver interfaces:[^341^]

> `DB` is a user-facing type. It's not an interface, but a concrete type (a struct) implemented in `database/sql` itself. It is backend-independent and encapsulates a lot of functionality that is common to all backends, like connection pooling.

The driver interfaces are minimal:

```go
type Driver interface {
    Open(name string) (Conn, error)
}

type Conn interface {
    Prepare(query string) (Stmt, error)
    Close() error
    Begin() (Tx, error)
}
```

Optional capabilities are added via extension interfaces:

```go
type ConnBeginTx interface {
    Conn
    BeginTx(ctx context.Context, opts TxOptions) (Tx, error)
}
```

Claim: If backends add optional capabilities, these can be selectively utilized in the backend-independent layer without exposing them directly to the user.
Source: Eli Bendersky / Design patterns in Go's database/sql package
URL: https://eli.thegreenplace.net/2019/design-patterns-in-gos-databasesql-package/
Date: 2019-03-27
Excerpt: "If backends add optional capabilities, these can be selectively utilized in the backend-independent layer without exposing them directly to the user."
Context: Deep analysis of database/sql design patterns
Confidence: high

#### 2.8.2 Go CDK Architecture

The Go CDK explicitly follows the database/sql pattern:[^713^]

> The portable APIs that the Go CDK exports (like `blob.Bucket` or `runtimevar.Variable`) are concrete types, not interfaces. ... Instead, we follow the example of `database/sql` and separate out the implementation-agnostic logic from the interface. The implementation-agnostic logic-containing concrete type is the portable type. We call the interface the driver.

Benefits:
1. Portable type can perform higher-level logic without complicating the driver interface.
2. Methods can be added to the portable type without breaking compatibility.
3. Optional driver interfaces are hidden from users via type assertions in the portable type.

#### 2.8.3 Implication for Mesh Adapter Generation

The Mesh architecture should follow this pattern:

```go
// User-facing portable type (concrete struct)
type Substrate struct {
    adapter Adapter
}

// Driver interface (minimal, for adapter implementers)
type Adapter interface {
    Create(ctx context.Context, req CreateRequest) (*CreateResponse, error)
    Read(ctx context.Context, req ReadRequest) (*ReadResponse, error)
    // ... core methods
}

// Optional extension interface
type FilesystemAdapter interface {
    Adapter
    ReadFile(ctx context.Context, path string) ([]byte, error)
    WriteFile(ctx context.Context, path string, data []byte) error
}
```

**Recommendation**: Adopt the portable type + driver interface pattern. The user-facing API is a concrete struct; the adapter interface is minimal and implementation-focused. Optional capabilities use extension interfaces.

### 2.9 Interface Embedding for Adapter Composition

#### 2.9.1 Embedding Interfaces in Structs

Go allows embedding an interface in a struct to get automatic method forwarding:[^683^]

```go
type StatsConn struct {
    net.Conn        // embedded interface
    BytesRead uint64
}

func (sc *StatsConn) Read(p []byte) (int, error) {
    n, err := sc.Conn.Read(p)
    sc.BytesRead += uint64(n)
    return n, err
}
```

This provides all methods of `net.Conn` automatically, while allowing selective override.

Claim: Embedding the interface gives us all these forwarding methods for free, and we can override just the ones we need.
Source: Eli Bendersky / Embedding in Go: Part 3 - interfaces in structs
URL: https://eli.thegreenplace.net/2020/embedding-in-go-part-3-interfaces-in-structs/
Date: 2020-08-19
Excerpt: "Embedding the interface gives us all these forwarding methods for free, and we can override just the ones we need."
Context: Detailed analysis of struct embedding patterns
Confidence: high

#### 2.9.2 Implication for Generated Adapters

If a generated adapter wraps an existing type that already implements most of the interface, embedding can reduce boilerplate. However, for AI-generated adapters that implement interfaces from scratch, explicit method declarations are clearer and self-documenting.

**Recommendation**: For generated adapters, prefer explicit method implementations over interface embedding. Embedding is more appropriate for decorator/wrapper adapters than for fresh implementations.

## 3. Contradictions and Conflict Zones

### 3.1 Wrap vs. Don't Wrap

There is genuine disagreement in the Go community about error wrapping:[^637^][^766^]

- **Go Team (Marcel van Lohuizen)**: "I do and don't... If I wanna have context, I wrap it. If I create a new error, I wrap it. But sometimes you're not really adding too much information, and then I don't."
- **Google Style Guide**: Translate at system boundaries rather than wrap.
- **Peter Bourgon**: "The risk of overwrapping... is much lower than the risk of underwrapping."
- **Rednafi**: "%v` as the conservative default` for libraries to avoid exposing implementation details.

**Resolution for adapters**: Use a hybrid approach. Wrap with `%w` when adding meaningful context within the adapter layer. Use `%v` (or translate to sentinel errors) when crossing the provider boundary so callers aren't coupled to provider-specific error types.

### 3.2 Optional Interfaces vs. ErrNotSupported

Some projects (notably early versions of certain Go libraries) used `ErrNotSupported` sentinel errors for unsupported operations. The Go team explicitly moved away from this toward extension interfaces for `io/fs` and `database/sql`.

However, `net/http.ErrNotSupported` exists as a sentinel in the standard library.[^711^] This suggests that sentinel errors for "not supported" are acceptable at the application level, while extension interfaces are preferred for library/plugin APIs.

**Resolution for adapters**: Use extension interfaces for adapter capabilities. Use `ErrNotSupported` only in the portable type when an optional method is called but the underlying adapter doesn't implement the extension interface.

### 3.3 Should Context Be Stored in Structs?

The context package docs say "Do not store Contexts inside a struct type."[^718^] But `net/http.Request` stores a context, and `database/sql.DB` stores connection state that is context-adjacent.

**Resolution for adapters**: Never store context in adapter structs. If an adapter needs to manage background goroutines, use a `quit` channel or a dedicated lifecycle context created during `Start()`, not a stored request context.

### 3.4 Go Generate vs. Build-Time Generation

Some argue that `go generate` should not be part of the build process because it can run arbitrary commands.[^638^] Others argue that checking generated files into version control is the standard practice.

**Resolution**: Follow the Go team's stated intent: `go generate` is for developers, not for end users building from source. Check generated files into version control. Run `go generate` in CI to verify they are current. Do NOT make `go generate` a dependency of `go build`.

## 4. Gaps in Available Information

1. **Crossplane provider patterns**: No authoritative documentation was found on how Crossplane providers handle optional interface methods in Go. The Crossplane ecosystem uses a different architecture (Kubernetes controllers) that doesn't directly map to Go interface patterns.

2. **AI-specific generation risks**: No literature was found on how AI-generated Go code specifically risks interface satisfaction failures or context leakage. The general Go literature applies, but AI-specific failure modes (hallucinated method names, incorrect receiver types) are undocumented.

3. **Conformance testing for generated adapters**: While Go CDK provides `drivertest` packages for conformance testing, there is no widely-adopted pattern for auto-generated adapter conformance tests. This is a gap the Mesh project could fill.

4. **Context cancellation in blocking provider calls**: There is limited guidance on how adapters should handle context cancellation when the underlying provider library has no context support and performs blocking I/O. The general advice is "check ctx.Done()" but this doesn't help with uninterruptible blocking calls.
